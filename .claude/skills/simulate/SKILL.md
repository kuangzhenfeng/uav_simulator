# Simulate

## 目标

用最少的构建和仿真次数获得可信结论。流程服务于修复，不用流程完整度代替代码进展。

遵守仓库 `CLAUDE.md`。不撤销用户已有修改，不主动提交代码。

## 默认参数

| 参数 | 默认值 |
|---|---|
| `mode` | `fix` |
| `focus` | 根据症状自动收窄 |
| `smoke_duration` | 20 秒 |
| `final_duration` | 60 秒 |
| `slomo` | 8 |
| `max_iterations` | 6 |
| `diagnosis_timebox` | 10 分钟 |

用户明确指定场景、时长、目标或预算时，以用户要求为准。

## 模式选择

| 模式 | 适用场景 | 是否需要修改前基线 |
|---|---|---|
| `run` | 只构建、测试或运行仿真 | 否 |
| `fix` | 明确报错、已知复现、局部行为缺陷 | 通常否 |
| `tune` | 性能调参、算法比较、指标优化 | 是 |
| `regression` | 验证控制链、安全层或共享行为改动 | 是 |
| `add_log` | 缺少定位问题所需的判决数据 | 否 |

默认选择最窄模式。不要因为调用了本 skill 就自动执行完整优化流程。

## 首轮决策

先完成以下只读预检，不构建、不仿真：

```powershell
git status --short
git diff -- <relevant-files>
rg -n "<symptom-or-tag>" Source Logs/uav.log
```

只读取与症状直接相关的代码和日志。优先读取 `Logs/uav.log`；只有过滤日志缺少关键上下文时才读取 `Logs/uav_full.log`。

然后按以下规则行动：

1. 编译错误、断言位置明确、明显逻辑错误或用户已提供可靠复现：直接形成假设并修改，不先跑基线。
2. 调参、性能优化、随机行为比较、安全改动或需要证明“没有退化”：先建立可比基线。
3. 根因不明：在 `diagnosis_timebox` 内收窄到最多两个候选；仍无证据时，添加一个最小结构化诊断点，不继续无界阅读。
4. 现有日志来自同一场景和参数，且相关代码自日志生成后未变化时，可复用为基线；不能确认时重新运行。

简单问题只需一行假设，不要求因果 DAG、候选评分表或全指标表。只有问题跨越两个以上控制模块时才写简短因果链。

## 高效闭环

### 1. 形成单一干预

记录四项即可：

- 假设：本次修改要证伪什么。
- 证据：对应代码或日志位置。
- 预期：哪个目标信号应改善。
- 否决：哪些硬失败或关键指标会导致回滚。

一次干预可以修改多个紧密相关位置，但必须共同服务于同一假设。先完成该干预的全部编辑，再构建；不要每改一行就构建。

### 2. 按风险验证

| 风险 | 变更范围 | 必需验证 |
|---|---|---|
| L0 | skill、文档、纯注释 | 不构建，不仿真 |
| L1 | 局部 C++、日志、明确 bug | 构建 + 20 秒 smoke；已有自动化测试直接覆盖时再运行测试 |
| L2 | NMPC、飞控、任务状态机、共享控制路径 | 构建 + 相关测试 + 20 秒 smoke；通过后运行一次完整仿真 |
| L3 | CBF、安全边界、动力学、性能调参、回归结论 | 修改前完整基线 + 构建/测试 + 候选完整仿真 |

所有 C++ 修改后至少运行一次仿真。先 smoke 再完整仿真；smoke 已出现硬失败时，不浪费时间运行完整仿真。

Windows 可用命令，按上表选择，不要机械地全部执行：

```powershell
Script\build.bat
Script\test.bat
Script\sim.bat <duration> <slomo>
```

macOS/Linux 使用对应 `.sh`。

测试规则：

- 当前 `Script/test.*` 运行完整 `UAVSimulator` 测试集，不支持目标过滤。
- L1 只有在现有自动化测试直接覆盖改动时才运行 `Script/test.*`。
- 修改公共控制契约、安全层或多个模块时运行 `Script/test.*`。
- 不要在修改前基线阶段默认运行完整测试。
- 源码未变化时，不重复已经成功的构建或测试。

### 3. 只分析相关信号

每次仿真先扫描硬失败，再提取本次目标相关 TAG：

```powershell
rg -n "Crash|penetration|collision|NaN|Inf|Overtime timeout|force completing|Attitude anomaly|NumericalFailure" Logs/uav.log Logs/uav_full.log
rg -n "<target-tags>" Logs/uav.log
```

不要为局部问题默认计算所有速度、姿态、Yaw、CBF 和 NMPC 指标。调参、安全、回归或 `add_log` 时才读取 [references/metrics.md](references/metrics.md)。

### 4. 接受、回滚或停止

接受候选需要满足：

- 目标现象改善或可靠消失。
- 没有新增硬失败。
- 变更范围与假设一致。
- L3 或 A/B 任务中，关键非目标指标没有明显退化。

失败时只精确撤销本轮由自己产生的修改，不使用 `git reset --hard`，不覆盖用户已有修改。

以下任一情况立即停止继续试错并报告证据：

- 连续两个候选没有可测改善。
- 下一候选缺少可区分结果的日志。
- 问题属于场景、资产、环境或外部依赖。
- 剩余预算不足以完成下一轮验证。

## 基线和 A/B

仅 `tune`、`regression` 和 L3 默认要求 A/B：

1. 使用相同场景、种子、时长和 `slomo`。
2. 保存基线日志副本，避免候选仿真覆盖。
3. 比较硬失败和目标相关指标，不生成无关指标。
4. 场景非确定且无固定种子时，不声称微小差异有效；重复关键候选或只接受明显改善。

## 子 Agent 规则

主流程默认不使用子 Agent。`rg` 和直接阅读能完成的分析由主 Agent 完成。

只有满足以下条件时才使用：

- 问题跨越两个以上相互独立的模块。
- 日志和代码扫描可以真正并行，预计能缩短总耗时。
- 子 Agent 只读，不修改文件、不构建、不运行仿真。

每个阶段最多一轮、每轮最多两个子 Agent。不要让不同子 Agent 重复扫描同一文件，也不要为了生成表格而调用子 Agent。

## `add_log`

只添加能够区分当前候选根因或判定硬门槛的日志。使用 `[TAG] key=value`、`LogUAVMetrics`、英文日志；多 Agent 携带 `Agent=%d`；高频路径使用 `UE_LOG_THROTTLE`；禁止 `Verbose`。

添加后按 L1 验证，并用 `rg "<NEW_TAG>" Logs/uav.log` 确认输出。详细 TAG 清单见 [references/metrics.md](references/metrics.md)。

## 最终报告

保持简短，包含：

1. 修改内容和根因。
2. 实际执行的构建、测试、smoke 或完整仿真。
3. 目标结果；只有执行 A/B 时才给基线对比。
4. 未解决风险或下一项最高价值实验。
