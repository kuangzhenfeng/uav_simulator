# Simulate Skill

## 概述

运行仿真并自主修复所有问题，直到无人机能稳定快速飞行且轨迹跟踪达标。

## 性能指标体系

指标基于各机型 MaxVelocity（AG-60=8m/s, AG-20=12m/s）按比例设定，而非固定绝对值。

> **指标是动态的，不是固定的。** 以下为初始目标值，经过多轮迭代仍未达标时，应按下方「指标降级策略」逐步调整。

### 速度指标

| 指标 | 达标 | 不达标 |
|------|------|--------|
| 巡航速度 | ≥ 50% MaxVel 持续稳定（AG-60: 4 m/s, AG-20: 6 m/s） | < 30% MaxVel 持续超过 15s |
| 速度恢复 | 避障降速后 8s 内恢复到 ≥ 40% MaxVel | 降速后 15s 仍未恢复 |
| 停止时间 | 单次 ≤ 5s（含避障急停恢复） | 单次 > 10s |
| 低速持续时间 | 单次 < 30% MaxVel 持续 ≤ 15s | 单次 > 30s 或频繁反复（≥ 3次） |

### 轨迹指标

| 指标 | 达标 | 不达标 |
|------|------|--------|
| 整体进度 | 仿真期间持续向目标推进，无长时间原地徘徊 | 原地徘徊 > 20s 或绕圈 |
| 轨迹跟踪偏差 | 所有 UAV 横向偏差 ≤ 5m | 存在 UAV 偏差 > 8m |

### 安全指标

| 指标 | 达标 | 不达标 |
|------|------|--------|
| 姿态稳定性 | Roll/Pitch 在 ±30° 以内，无异常震荡 | Roll/Pitch > 60° 或控制输出剧烈震荡 |
| 避障安全 | 无碰撞（允许 NearDist < SafeDistance 的接近事件） | 发生碰撞穿透 |
| 避障轨迹 | 绕障路径流畅，能通过障碍群 | 被障碍群完全阻挡或原地徘徊 |

> 场景说明：障碍物为 1m 直径球体、数百个密集分布在航线上。要求 UAV 能以合理速度穿越障碍群，而非追求最大速度。
> 如用户指定了具体目标速度或航线，以用户要求为准，覆盖以上默认值。

### 指标降级策略

当同一指标连续 **3 轮**迭代仍未达标时，执行降级：

1. **确认已达实际瓶颈**：检查是否已尝试过该指标的常见修复方向，确认非简单 bug 导致
2. **降级指标**：将该项指标放宽到当前实际能达到的最佳水平，建立新的合理基线
3. **记录降级**：在迭代记录中明确标注指标降级，记录原始值、降级后的值、降级原因

降级幅度参考（每级约降低 20%）：

| 指标 | 初始值 | 降一级 | 降二级 |
|------|--------|--------|--------|
| 巡航速度达标线 | 50% MaxVel | 40% MaxVel | 30% MaxVel |
| 速度恢复达标线 | 40% MaxVel / 8s | 30% MaxVel / 10s | 25% MaxVel / 12s |
| 停止时间上限 | 5s | 8s | 12s |
| 低速持续上限 | 15s | 20s | 30s |
| 轨迹偏差上限 | 5m | 7m | 10m |
| 徘徊判定阈值 | 20s | 30s | 45s |

**降级原则：**
- 安全指标（碰撞、姿态稳定性）**不降级**，安全是硬约束
- 速度和轨迹指标可按需降级
- 每项指标独立降级，不捆绑
- 降级后继续迭代，以新指标为基准评估

## 执行流程

### 1. 并行准备

同时进行以下操作：

- **后台运行仿真**（`run_in_background: true`）：
  - Windows: `Script\sim.bat 80`
  - macOS/Linux: `Script/sim.sh 80`
- **读取核心代码**，重点关注与速度/运动/避障相关的逻辑：
  - `Source\uav_simulator\Core\UAVPawn.cpp/h` — 飞行动力学与速度控制
  - `Source\uav_simulator\Planning\PathPlanner.cpp/h` — 路径规划与速度指令
  - `Source\uav_simulator\Planning\NMPCAvoidance.cpp/h` — 避障控制与速度约束
  - `Source\uav_simulator\Planning\ObstacleManager.cpp/h` — 障碍物管理
  - 其他与飞行控制、速度限制相关的文件

### 2. 读取日志并分析

等待仿真完成后读取 `Logs/uav.log`，重点提取：

- **速度数据**：各 UAV 的实时速度、各 Agent 型号 MaxVelocity、平均速度占 MaxVel 百分比
- **停止事件**：速度接近 0 的时间点和持续时长
- **低速事件**：速度 < 30% MaxVel 的时间点和持续时长
- **速度恢复**：降速后恢复到 40% MaxVel 的耗时
- **位置变化**：是否有持续位移，还是原地不动
- **姿态数据**：Roll/Pitch 是否超出 ±30°
- **碰撞事件**：是否有障碍物穿透
- **轨迹偏差**：各 UAV 横向偏差是否 ≤ 5m
- **全局重规划次数**：频繁重规划（> 3次）表明局部规划器严重卡死

### 3. 对照判定标准评估

按 Agent 型号分别计算达标线，逐条检查指标表，标记每项为"达标"或"不达标"。

### 4. 若不达标 → 定位根因并修复

根据不达标项，定位根本原因：

| 不达标现象 | 常见根因 |
|-----------|---------|
| 巡航速度过低 | 轨迹 MaxVel 与机型不匹配、EMA 平滑过度、自适应时间缩放冻结轨迹 |
| 长时间停止 | 避障过于保守、安全距离过大、NeedsCorrection 保持过长 |
| 速度不恢复 | 缺少速度恢复逻辑、NMPC 温启动陷入零控制循环、卡死检测过于激进 |
| 原地徘徊 | 路径规划死锁、目标点不可达、避障方向选择错误 |
| 姿态震荡 | 控制器增益过大、阻尼不足、EMA 平滑导致控制延迟 |
| 频繁重规划 | 卡死检测阈值过低、逃逸逻辑不足、NMPC 在密集障碍区求解失败 |
| 碰撞 | 安全距离过小、紧急制动阈值不当、CBF-QP 约束不足 |
| 轨迹偏差过大 | 偏差保护阈值不当、参考跟踪权重过低、前馈控制不足 |

**诊断日志：** 如果日志信息不足以定位根因，应主动在相关代码中添加临时诊断日志，帮助分析问题。添加日志时遵循以下原则：

- 使用 `UE_LOG` 或已有的日志宏，日志等级用 `Warning`（确保能被捕获）
- 如果诊断日志输出频率较高，必须使用 `UE_LOG_THROTTLE` 限制输出频率
- 诊断日志应包含关键字段（如当前速度、目标速度、控制输出、避障状态等），以便精确判断问题所在
- 日志用英文，注释用中文
- 修复问题后，如果诊断日志不再需要，应及时清理或降级

修复步骤：
1. 读取相关源码，定位具体代码位置
2. 必要时添加诊断日志
3. 实施修复
4. 编译：Windows `Script\build.bat` / macOS Linux `Script/build.sh`
5. 编译失败 → 修复编译错误 → 重新编译
6. 编译成功 → 回到步骤 1

### 5. 每次迭代记录

记录本轮：
- 修改了什么文件、哪些代码
- 为什么修改（对应哪个不达标项）
- 修改前后的关键数据对比

### 6. 全部达标 → 输出总结

当所有指标达标时，输出：
- 各 Agent 各项指标的最终数据
- 总迭代次数和修改摘要
- 确认飞行稳定快速、轨迹跟踪达标

## 约束

- 不要在迭代之间询问确认，保持自主运行
