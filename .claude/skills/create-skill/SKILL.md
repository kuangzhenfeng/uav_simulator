# Create Skill（创建 Skill）

## 概述

用于从需求出发创建一个通用、可复用的 skill。核心原则：**skill 不是笔记，而是可指导他人独立完成同类任务的行动手册** — 必须包含前置条件、执行流程、判断规则和验证方式。

## 执行流程

### 1. 明确需求

确定 skill 要解决什么问题：

- **目标**：skill 要帮助完成什么类型的任务
- **使用者**：谁会用这个 skill（自己、团队成员、AI 助手）
- **输入**：skill 使用者需要提供什么信息/参数
- **输出**：skill 执行后应产出什么结果

> 如果需求不够明确，先向使用者确认再继续。

### 2. 调研现有 skill

检查 `.claude/skills/` 下是否已有相关 skill：

| 情况 | 处理方式 |
|------|---------|
| 已有 skill 完全覆盖需求 | 不新建，直接使用已有 skill |
| 已有 skill 部分覆盖，但缺少关键内容 | 更新已有 skill，补充缺失部分 |
| 已有 skill 领域完全不同 | 新建 skill |

> **优先更新已有 skill，仅当领域完全不同才新建。**

### 3. 设计结构

根据需求确定 skill 的章节结构。参考以下骨架，按需裁剪：

| 章节 | 是否必需 | 作用 |
|------|---------|------|
| 概述 | 必需 | 一句话说明核心原则 |
| 执行流程 | 必需 | 按步骤编号，每步说明做什么、怎么判断、产出什么 |
| 判断规则/决策表 | 按需 | 用表格列出常见判断点及对应处理 |
| 常见误区 | 按需 | 列出容易犯的错误及正确做法 |
| 示例/模板 | 按需 | 提供输出格式模板或代码模板 |
| 先做项目发现 | 按需 | 当 skill 需要适配不同项目时使用 |

> 不要为了"完整性"强行加上不需要的章节。内容空洞的章节比缺失章节更糟。

### 4. 编写内容

逐章编写，遵循以下原则：

**4.1 概述**
- 一句话核心原则，说明 skill 的价值主张
- 不写背景铺垫，直接给出原则

**4.2 执行流程**
- 按步骤编号（1, 2, 3...），不超过 6 步
- 每步包含：做什么 → 怎么判断 → 产出什么
- 判断点用表格呈现
- 步骤顺序应遵循实际执行顺序

**4.3 判断规则**
- 用 Markdown 表格，不要用散列文本
- 每行一个明确的场景 → 对应的处理方式
- 覆盖常见场景，不追求穷举

**4.4 常见误区**
- 每条误区包含：错误做法 → 正确做法
- 只列真正容易犯的错，不列显而易见的

**4.5 示例/模板**
- 提供可直接复制使用的格式模板
- 模板中的占位符用 `[描述]` 标注，不写具体值

### 5. 通用化审查

对编写完成的 skill 进行通用化审查，参照 `generalizing-skills` skill 的规则：

| 类型 | 特征 | 处理方式 |
|------|------|----------|
| 业务专有名词 | 产品名、系统名、模块名 | 替换为通用占位描述 |
| 特定技术栈 | 仅某框架/库的专属 API | 改为"按需扩展"说明 |
| 具体示例值 | 特定路径、URL、配置值 | 保留为参考示例，去掉业务语义注释 |
| 业务语义注释 | "Cloud专属"、"生产路径" | 改为通用功能描述 |
| 特定场景变体 | 只在某业务场景出现的类/组件 | 改为扩展示例，不作为核心规范 |

> 判断标准：**"如果换一个完全不同的业务场景，这条内容还适用吗？"**
> - 适用 → 保留
> - 不适用但可抽象 → 改写为通用原则
> - 不适用且无法抽象 → 删除

### 6. 确定 skill 元信息与写入

| 字段 | 规则 |
|------|------|
| `name` | 从 skill 核心目标提炼，用短横线分隔，简洁明确（如 `create-skill`、`protocol-adaptation`） |
| `description` | 一句话描述何时使用，格式："当...时使用" |
| `文件名` | 与 `name` 一致，目录下放 `SKILL.md` |

写入路径：`.claude/skills/<name>/SKILL.md`

写入后，用一个与原需求完全不同的场景测试 skill，看是否能正确指导。若验证失败，回溯步骤 4~5 修正。

## Skill 格式规范

- **frontmatter**：必须包含 `name` 和 `description`
- **标题层级**：`#` 为 skill 名称，`##` 为主要章节，`###` 为子章节
- **表格**：用 Markdown 表格呈现判断规则、决策点、对比等结构化信息
- **代码块**：按需提供模板或示例，用语法高亮标记
- **中文**：skill 内容用中文编写，与项目注释风格一致

## 判断规则

### Skill 粒度

| 情况 | 粒度选择 |
|------|---------|
| 目标是一个完整独立流程 | 一个 skill |
| 目标包含多个独立子流程，各自可复用 | 多个 skill，每个对应一个子流程 |
| 步骤太多、太细 | 合并为关键阶段，不超过 6 步 |

### 是否值得创建 skill

> 问自己三个问题：
> 1. **频率**：这类任务以后会出现至少 3 次以上吗？
> 2. **可迁移**：去掉业务细节后，流程和规则还能指导他人完成类似任务吗？
> 3. **完整性**：skill 是否包含足够的前置条件、判断标准和验证方式，让使用者能独立完成？

三个问题都回答"是" → 创建 skill
任一回答"否" → 不创建，仅在本次对话中处理

## 常见误区

| 误区 | 正确做法 |
|------|---------|
| 把 skill 写成个人笔记，只有自己能看懂 | skill 是行动手册，必须让他人能独立执行 |
| 照抄现有 skill 的结构而不适配需求 | 按需裁剪章节，内容空洞的章节比缺失章节更糟 |
| skill 写得太抽象，失去指导价值 | 保留具体的判断标准、决策表和验证方式 |
| skill 写得太具体，耦合了业务场景 | 用通用化规则剥离业务专有内容 |
| 没有验证 skill 的可用性 | 用不同场景测试 skill 是否能正确指导 |
| 与已有 skill 重叠时新建而不合并 | 优先更新已有 skill；仅当领域完全不同才新建 |
| skill 名称太长或太模糊 | 名称应简洁且明确反映核心目标 |
