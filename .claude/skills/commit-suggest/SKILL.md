# Commit Suggest（提交信息建议）

## 概述

分析当前未提交的代码变更，参考项目历史提交风格，生成一条简短的 commit message。**仅输出建议的提交信息，不要有多余输出。**

## 执行流程

### 1. 采集历史提交风格

从 `git log` 中提取项目的 commit message 风格特征：

- **emoji 前缀**：确定使用的 emoji 及其含义（🐞 fix / 🦄 refactor / ✨ feat / 📃 docs / 🔧 build）
- **格式结构**：`emoji type: 描述内容` 或其他固定格式
- **描述语言**：中文或英文
- **描述粒度**：简要概括还是详细列举多个改动点

### 2. 采集未提交变更

执行 `git diff` 和 `git diff --cached` 获取未提交的代码变更，重点关注：

- 变更的文件和模块
- 变更类型（新增、修改、删除、重命名）
- 核心改动内容

### 3. 分类变更类型

| 变更特征 | 类型 | emoji |
|---------|------|-------|
| 修复 bug、修复异常行为 | fix | 🐞 |
| 重构代码结构、优化实现、移除无用代码 | refactor | 🦄 |
| 新增功能、新增模块 | feat | ✨ |
| 文档更新、注释更新 | docs | 📃 |
| 构建配置、版本号变更 | build | 🔧 |

当变更涉及多种类型时，选择**最核心变更**对应的类型。

### 4. 生成提交信息

按以下规则生成：

- **格式**：`{emoji} {type}: {描述}`
- **描述**：用中文，简洁概括核心改动
- **多改动时**：用中文顿号 `、` 分隔，列举关键改动点
- **长度**：控制在 80 字以内，不换行
- **语气**：陈述式，不加"了"、"的"等冗余字

### 5. 输出

**仅输出一行提交信息，不要有任何其他输出。** 不要加说明文字、不要加 markdown 格式、不要加引号。

## 判断规则

| 情况 | 处理 |
|------|------|
| 工作区和暂存区都有变更 | 合并分析，生成一条提交信息 |
| 仅删除文件 | 按删除所属的业务模块归类 |
| 变更跨多个模块 | 按核心改动归类，描述中列举关键模块 |
| 无法确定类型 | 默认归类为 refactor |

## 常见误区

| 误区 | 正确做法 |
|------|---------|
| 输出多条候选提交信息让用户选择 | 只输出一条最佳建议 |
| 输出分析过程和解释 | 仅输出最终的提交信息 |
| 提交信息过于笼统（"更新代码"） | 概括到具体的模块或功能点 |
| 提交信息过于冗长（超过 80 字） | 压缩到关键改动点 |
